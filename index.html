Can You Share Those Slides?

Using Quarto and revealjs for Efficient Presentations

Joe Corra

2026-01-25

Introduction: The Problem

Don’t let your slides be your only document!

Problems with presentation-only slides:

  • Slides lack context after the presentation
  • Recreating content is tedious

Problems with hybrid presentation-documentation slides:

  • Presentation suffers so slides can serve as documents
  • “Wall of text” slides are hard to read in any setting

Bad Slides: An Example

  • This slide performs two jobs poorly.

  • Bad for viewing (Detracts from presentation)

  • Bad for reading (just share a Word doc!)

What is Quarto?

Modern publishing system for technical content

  • Successor to R Markdown
  • Execute code in your document
  • Output to various formats (HTML, Word, PDF, & more)
  • Create documents, presentations, dashboards, websites, & books
  • Reproducible content in R, Python, Julia, & Observable
  • Works with RStudio, Positron, & Jupyter notebooks

What is revealjs?

HTML presentation framework

  • Web-based slides
  • Professional appearance
  • Interactive features
  • Works anywhere

Our Solution: One Script for Two Outputs

Use conditional content to control what appears where

::: {.content-visible when-format="revealjs"}
Brief slide content
:::

::: {.content-visible unless-format="revealjs"}
Detailed documentation
:::

Setting Up Your Document

YAML header:

---
title: "Your Presentation"
format:
  revealjs:
    theme: default
  html:
    toc: true
---

Rendering Your Outputs

Terminal commands for precise control:

# Slides for presenting
quarto render doc.qmd --to revealjs \
  --output slides.html

# Document for sharing
quarto render doc.qmd --to html \
  --output document.html

Or just comment out one of the formats in the YAML:

format:
  revealjs:
    theme: moon
    slide-number: true
    transition: slide
  # html:
  #   theme: lux
  #   toc: true
  #   number-sections: true

Best Practices

Tips for success:

  • Keep slide content minimal
  • Put details in document sections
  • Test both outputs regularly
  • Include code examples in both
